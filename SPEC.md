# 꼬꼬무AI SPEC

## 1. 제품 목표

`꼬꼬무AI`는 2022 개정 교육과정을 기준선으로 삼아, 선생님이 수업 주제별 챗봇을 만들고 학생이 공유 링크로 접속해 질문형 대화를 할 수 있는 웹앱이다.

이 앱의 핵심 차별점은 일반 AI 챗봇처럼 정답을 바로 제공하지 않는 것이다. 학생의 질문을 바탕으로 사고를 확장하고, 학생이 스스로 개념을 정리하도록 돕는다.

브라우저 탭 제목은 `꼬꼬무AI`로 표시한다. 첫 화면의 큰 제목은 `꼬리에 꼬리를 무는 AI`이며, 설명 문구는 `질문과 대화를 통해 스스로 알아가는, 여러분을 위한 공간입니다.`를 사용한다.

## 2. MVP 범위

MVP는 다음 범위로 제한한다.

- 교사 회원가입, 로그인, 승인 대기, 관리자 승인
- 교사 대시보드
- 챗봇 생성, 수정, 삭제, 공유
- 2022 개정 교육과정 기반 관련 영역/성취기준 추천
- 성취기준 추천 카드 다중 선택과 관련성 표시
- 학생용 공유 링크 접속
- 학생 가입 없는 로컬 대화
- 질문형 AI 대화
- 답변 Markdown 기본 렌더링
- 학생 대화 PDF/TXT 다운로드
- 교사별 사용량 확인
- 관리자 대시보드

현재 운영 목표 버전은 React/Vite/TypeScript 프런트엔드, Vercel 서버리스 API, Firebase Authentication, Firestore를 기준으로 한다.

로컬 개발 환경에서는 같은 도메인 규칙을 로컬 API 서버와 JSON 저장소로 검증하되, 배포 환경에서는 Firebase ID token과 Firestore 저장소를 사용한다.

브라우저용 Firebase Web SDK 설정이 있더라도 `VITE_FIREBASE_AUTH_ENABLED=true`가 설정된 운영 환경에서만 교사 Firebase Auth 화면이 활성화된다. 로컬 검증 환경은 이 플래그를 생략해 자동 승인 로컬 교사 흐름을 유지할 수 있다.

MVP에서 제외한다.

- 학생 계정
- 학생 대화 서버 저장
- 학생별 학습 분석
- 자동 채점
- 이미지/파일/음성 업로드
- 다국어 지원
- 공개 챗봇 갤러리
- 다른 교사의 챗봇 복제

## 3. 사용자 역할

### 3.1 학생

- 가입하지 않는다.
- 공유 링크로 접속한다.
- 이름, 학번, 이메일, 전화번호, 학교명, 학급 정보를 입력하지 않는다.
- 대화 기록은 학생 기기 브라우저에만 저장한다.
- 같은 기기와 같은 브라우저에서만 이전 대화를 이어갈 수 있다.
- 대화 기록을 PDF 또는 TXT로 내려받을 수 있다.

### 3.2 교사

- 이메일/비밀번호 또는 Google OAuth로 가입한다.
- 가입 시 실명과 학교 정보를 입력한다.
- 가입 후 관리자 승인 전까지 교사 기능을 사용할 수 없다.
- 승인 후 챗봇을 만들고 공유할 수 있다.
- 대시보드에서 자신의 챗봇, 공유 상태, 사용량, 오류 상태를 확인한다.
- 대시보드에서 표시 이름을 설정할 수 있다.

### 3.3 관리자

- 1차 관리자는 서비스 운영자인 사용자 본인이다.
- 가입 대기 교사를 승인하거나 거절할 수 있다.
- 교사 목록과 사용량을 확인한다.
- 문제 챗봇을 비활성화할 수 있다.
- 교사 계정에 비밀번호 재설정 메일을 발송할 수 있다.
- 관리자 작업 로그를 확인한다.
- 기본 모드에서는 학생 정보와 학생 대화 내용을 볼 수 없다.

## 4. 개인정보와 심의 대응 원칙

### 4.1 기본 모드

기본 모드는 학습지원 소프트웨어 심의 제외 가능성을 최대화하는 구조로 설계한다.

- 학생 개인정보를 수집하지 않는다.
- 학생 대화 기록을 서버에 저장하지 않는다.
- 학생 메시지는 AI 전송 전에 개인정보 탐지 필터를 거친다.
- 개인정보로 보이는 내용은 AI로 보내지 않고 다시 쓰도록 안내한다.
- 차단된 원문은 서버에 저장하지 않는다.

### 4.2 학생 개인정보 필터

MVP에서 우선 차단할 항목은 다음과 같다.

- 전화번호
- 이메일
- 주소
- 학번처럼 보이는 숫자 패턴

이름은 오탐 가능성이 높으므로 강제 차단보다 안내 중심으로 처리한다.

### 4.3 향후 제출 모드

향후 학습지원 소프트웨어 심의와 개인정보 처리 요건을 충족한 뒤 “제출 모드”를 추가할 수 있다.

제출 모드에서는 교사가 제출 식별 방식을 선택한다.

- 식별 정보 없음
- 제출 코드
- 번호
- 이름
- 학번
- 이름+학번

기본 추천값은 `제출 코드` 또는 `번호`이다. `이름+학번`은 학교 심의와 개인정보 안내가 갖춰진 경우에만 사용하도록 경고 문구를 표시한다.

## 5. 교사 가입과 승인

### 5.1 가입 방식

Firebase Authentication을 사용한다.

- 이메일/비밀번호 가입
- Google OAuth 가입
- 비밀번호 찾기

Google OAuth 가입도 최초 로그인 후 추가 정보 입력 화면을 거친다.

### 5.2 교사 가입 정보

교사는 가입 시 다음 정보를 입력하거나 선택한다.

- 실명
- 이메일
- 로그인 방식
- 학교 정보

학교 정보는 NEIS 학교기본정보 API를 사용한다.

- 학교명을 일부 입력하면 후보 목록을 보여준다.
- 교사는 목록에서 학교를 선택해야 한다.
- 저장 항목은 학교명, 학교종류, 시도교육청 코드, 표준학교 코드, 지역/주소 일부 정도로 제한한다.

### 5.3 교사 이름과 표시 이름

- `realName`: 가입 시 입력한 실명, 관리자 승인과 계정 관리용
- `displayName`: 교사 대시보드에서 설정하는 표시 이름

학생 화면에는 실명을 자동으로 노출하지 않는다. 교사가 설정한 표시 이름만 표시한다. 교사는 챗봇별로 교사 이름 표시를 끌 수 있다.

### 5.4 승인 상태

교사 계정은 다음 상태를 가진다.

- `pending`: 가입 완료, 관리자 승인 대기
- `approved`: 승인 완료
- `rejected`: 승인 거절
- `disabled`: 사용 중지
- `admin`: 관리자 권한

관리자 승인 대기 화면에는 이름, 이메일, 학교명, 학교급, 지역, 가입 방식, 가입일을 표시한다. 관리자는 학교명, 지역, 가입일로 필터링하고 여러 명을 선택해 일괄 승인 또는 일괄 거절할 수 있다. 거절 시 사유를 선택하거나 입력할 수 있다.

운영 최초 관리자 계정은 서버 환경변수 `KKOKKOMU_ADMIN_EMAILS`에 등록된 Firebase 계정 이메일로 부트스트랩한다. 해당 이메일로 로그인한 사용자가 NEIS 학교 검색 결과를 선택해 프로필을 제출하면 서버는 Firebase 토큰의 이메일을 기준으로 `admin` 상태를 부여한다. 가입 요청 본문에 담긴 이메일은 관리자 승격 판단에 사용하지 않는다.

## 6. 교사 대시보드

교사 대시보드 첫 화면은 `내 챗봇 목록` 중심으로 구성한다.

상단에는 다음 정보를 작게 보여준다.

- 새 챗봇 만들기
- 이번 달 사용량 요약
- 오류 알림

챗봇 카드에는 다음 정보를 표시한다.

- 챗봇 이름
- 수업 주제
- 학교급/학년군 또는 학년
- 공유 상태
- 최근 수정일
- 바로가기 버튼

챗봇 카드의 바로가기 버튼은 다음과 같다.

- 수정
- 미리보기
- 공유 링크
- 기록 다운로드 안내
- 삭제

공유가 꺼져 있으면 `공유 켜기`로 표시한다. 공유가 켜져 있으면 `링크 복사`와 `공유 끄기`를 제공한다.

`링크 복사` 결과는 화면 상단 알림이 아니라 해당 챗봇 항목 아래에 잠깐 표시한 뒤 자동으로 사라지게 한다. 사용자가 방금 누른 위치 근처에서 결과를 확인할 수 있어야 한다.

챗봇 삭제는 즉시 실행하지 않는다. 삭제 버튼 근처에 짧은 확인 메시지를 보여주고, 사용자가 확인했을 때만 삭제한다. 삭제된 챗봇은 즉시 목록에서 사라져야 한다.

교사는 챗봇 목록에서 여러 챗봇을 선택할 수 있다. `전체 선택`과 `선택 삭제` 기능을 제공하고, 여러 항목 삭제도 확인 후 실행한다.

## 7. 챗봇 생성

### 7.1 화면 구조

챗봇 생성 화면은 한 페이지 폼으로 만든다. 단계형 마법사처럼 화면을 강하게 나누지 않는다.

섹션은 다음과 같다.

- 기본 정보
- 관련 교육과정
- 대화 방식
- 공유 설정

교사는 위에서 아래로 자연스럽게 입력하되, 언제든 이전 항목으로 돌아가 수정할 수 있다.

### 7.2 필수와 선택

챗봇 생성 시 다음 항목을 입력하거나 설정한다.

- 챗봇 이름
- 학교급
- 학년군/학년
- 수업 주제
- 대화 목표
- 힌트 강도
- 페르소나
- 공유 설정

성취기준 선택은 권장하지만 필수는 아니다.

### 7.3 학교급과 학년

교사는 `학교급 + 학년군/학년`을 설정한다.

- 초등학교: 1-2학년군, 3-4학년군, 5-6학년군
- 중학교: 학년 선택
- 고등학교: 학년 선택
- 직업계고: 고등학교 범위 안에서 직업계고 맥락을 별도로 선택 가능

이 정보는 교육과정 추천, 학생용 말투, 질문 난이도, 힌트 수준 조절에 사용한다.

주제만 입력해도 추천을 받을 수 있다. 이 경우 앱은 여러 학교급/학년 후보를 함께 추천하고, “학교급과 학년을 선택하면 더 정확하게 추천할 수 있어요”라고 안내한다.

특수교육 관련 교육과정도 추천 검색 대상에 포함할 수 있다. 다만 UI의 학교급 표기는 학생과 교사가 이해하기 쉬운 한국어 라벨을 우선 사용한다.

### 7.4 수업 주제 입력

수업 주제는 챗봇의 중심이다.

교사가 너무 넓은 주제를 입력하면 앱은 바로 챗봇 생성을 진행하지 않는다. “어떤 단원의 어떤 개념이나 내용에 대해 챗봇을 만들고 싶으신가요?”라고 묻고 후보를 제안한다.

예:

- 품사의 종류와 특성 이해
- 관형사와 부사의 차이 비교
- 문장에서 조사와 어미 구분하기
- 문학 작품 속 표현 효과 찾기
- 직접 입력

폼에 보이는 예시 문구는 실제 기본값이 아니라 placeholder로만 표시한다. 교사가 입력하지 않은 값을 고정 데이터처럼 저장하거나 챗봇 생성에 반영하면 안 된다.

교사가 너무 좁거나 애매한 주제를 입력하면 긴 설명을 요구하지 않는다. 먼저 가능한 해석 후보를 2-3개 제안한다. 그래도 판단이 어려울 때만 한 문장 보완을 요청한다.

예:

> “프린트 2번”만으로는 수업 내용을 알기 어려워요. 혹시 아래 중에 가까운 내용이 있나요?

### 7.5 대화 목표

대화 목표는 필수 항목이다.

수업 주제를 입력하면 앱이 대화 목표 초안을 자동 추천한다. 교사는 추천 목표를 그대로 쓰거나 수정할 수 있다.

예:

> 학생이 문장에서 품사의 쓰임을 살펴보고, 관형사와 부사의 차이를 스스로 설명하도록 돕는다.

## 8. 교육과정 추천

### 8.1 추천 원칙

교육과정 추천은 챗봇 생성의 핵심 단계가 아니라, 수업 주제 입력 후 나타나는 “관련 교육과정” 보조 영역으로 둔다.

교사는 추천된 영역/성취기준을 확인하고 그대로 두거나 수정할 수 있다. 성취기준 선택 없이도 챗봇 생성은 가능하다.

### 8.2 추천 방식

정리된 2022 개정 교육과정 Markdown에서 성취기준 단위 chunk를 만들고 검색 인덱스로 구성한다.

추천 흐름은 다음과 같다.

1. 교사가 수업 주제를 입력한다.
2. 서버가 관련 성취기준 후보를 검색한다.
3. AI가 검색된 후보 안에서만 상위 후보를 재랭킹한다.
4. 앱은 관련 교육과정 보조 영역에 추천 결과를 보여준다.

AI는 검색 후보 밖의 성취기준을 만들어내면 안 된다.

### 8.3 추천 UI

기본 화면에는 과목/영역/성취기준을 카드 형태로 보여준다.

교사는 다음 선택을 할 수 있다.

- 성취기준 선택
- 선택 해제
- 자세히 보기
- 더 보기
- 접기

추천 카드 선택은 필수가 아니다. 교사가 아무 성취기준도 선택하지 않으면 가장 관련성이 높은 성취기준 1개를 자동으로 반영한다. 교사가 여러 개를 선택하면 선택된 성취기준을 모두 챗봇 맥락에 포함한다.

기본으로 3개 카드를 보여주고, 추가 후보는 `더 보기`로 펼친다. `더 보기` 버튼은 추천 안내 문구와 같은 줄의 우측에 배치한다.

추천 라벨은 숫자 점수가 아니라 관련성 등급으로 표시한다.

- 관련성 상
- 관련성 중
- 관련성 하

카드에는 성취기준 번호를 반드시 표시한다. 교사가 실제 교육과정 문서와 대조하기 쉬워야 한다.

라벨 아래에는 추천 이유를 한 문장으로 적는다. 선택 버튼은 카드 맨 윗줄 우측에 배치해 성취기준 번호와 관련성 표시를 가리지 않도록 한다.

### 8.4 교사용 근거 표시

교사 화면에는 내부 Markdown 파일명이나 경로를 보여주지 않는다.

기본 표시:

- 학교급/학년군
- 과목
- 영역
- 성취기준 번호
- 성취기준 한 줄 요약
- 관련성 상/중/하
- 추천 이유 한 문장
- 출처: `2022 개정 ○○과 교육과정 [별책X]`

`자세히 보기`를 누를 때만 다음 정보를 펼쳐 보여준다.

- 성취기준 원문 전체
- 관련 원문 일부
- 주변 문맥
- 페이지/섹션
- 원문 출처 세부 정보

학생 화면에는 교육과정 근거를 매번 보여주지 않는다. 필요한 경우 “오늘의 수업 범위”만 짧게 표시한다.

## 9. 챗봇 대화 정책

### 9.1 핵심 원칙

챗봇은 정답을 직접 제공하지 않는다.

학생의 질문을 바탕으로 다음 사고 단계로 이끈다. 교사가 입력한 페르소나보다 다음 고정 정책이 우선한다.

- 정답 직접 제공 금지
- 질문 중심 대화
- 범위 밖 질문 차단
- 개인정보 보호
- 편안한 존댓말
- 학교급에 맞는 표현
- 프롬프트 인젝션 방어

### 9.2 교육과정과 지식 사용

교육과정은 답변의 기준선과 경계선으로 사용한다.

AI는 그 범위 안에서 학생 수준에 맞는 교과 설명, 비유, 예시, 사고 질문을 생성할 수 있다. 다만 선택된 학교급, 과목, 주제, 성취기준을 넘는 내용은 설명하지 않고, 필요한 경우 교사에게 확인하도록 안내한다.

교육과정 원문은 “무엇을 가르칠 것인가”를 정하고, AI는 “어떻게 질문하며 도울 것인가”를 담당한다.

### 9.3 시작 멘트

챗봇은 학생 화면이 열리면 먼저 짧은 시작 멘트를 보여준다. 이 멘트는 정답이나 설명이 아니라, 학생이 무엇을 물어보면 좋은지 안내하는 역할이다.

시작 멘트에는 다음을 담는다.

- 수업 주제
- 궁금한 점을 입력하라는 안내
- 함께 생각해 보자는 방향

말투는 학교급에 맞춘 편안한 존댓말로 한다.

예:

> 오늘은 품사의 종류와 특성에 대해 함께 알아볼 거예요. 궁금한 점을 적어 주세요. 바로 답을 찾기보다, 같이 생각해 보면서 알아가요.

### 9.4 말투

학생에게는 반말을 쓰지 않는다. 하지만 과한 높임말이나 딱딱한 행정 문구도 쓰지 않는다.

- 초등: 쉽고 따뜻하게
- 중등: 간결하고 친근하게
- 고등: 차분하고 자연스럽게

채팅 답변, 시작 멘트, 개인정보 안내, 범위 밖 질문 안내, 다운로드 안내, 오류 메시지 모두 같은 원칙을 따른다.

### 9.5 한 번에 하나의 질문

챗봇은 한 번에 질문을 하나만 던진다.

한 응답에는 핵심 질문 하나와 필요 시 짧은 단서 하나만 제공한다. 여러 질문을 던져야 할 때도 순서대로 하나씩 나눈다.

### 9.6 나선형 이해 구조

학생이 답하면 이전 학생 질문과 답변 흐름을 짧게 정리하고, 그 위에 다음 질문을 얹는다.

매 응답마다 길게 요약하지는 않는다. 짧은 응답에서는 학생의 말 한 부분만 되짚고 바로 다음 질문으로 간다.

3-5턴마다 한 번씩 “지금까지 생각 정리”를 제공한다. 학생이 “정리해줘”라고 요청하면 언제든 현재까지의 생각을 요약한다.

요약은 다음 구조를 따른다.

- 지금까지 생각한 것
- 아직 확인할 점
- 다음에 생각할 질문

정답을 대신 써 주거나 과제 답안처럼 완성해 주지 않는다.

### 9.7 힌트 강도

교사는 챗봇별로 힌트 강도를 선택한다.

- 낮음: 거의 질문만 한다.
- 보통: 질문에 짧은 단서를 붙인다.
- 높음: 짧은 부분 설명을 제공한 뒤 다음 사고 질문으로 연결한다.

힌트 강도가 높아도 최종 정답을 직접 말하지 않는다.

### 9.8 학생이 막혔을 때

학생이 “모르겠어요”, “어려워요”처럼 막히면 챗봇은 질문을 더 작게 쪼갠다.

추상적인 질문을 줄이고 관찰, 선택, 비교처럼 저차원의 사고 활동으로 낮춘다. 학교급에 맞게 쉬운 표현으로 다시 말한다.

### 9.9 오답 반응

학생이 명백히 틀린 답을 해도 바로 정정하지 않는다.

먼저 학생 생각 중 살릴 수 있는 부분을 짚고, 비교, 반례, 확인 질문으로 다시 생각하게 한다. 필요하면 짧은 단서를 제공하지만 정답 문장으로 끝내지 않는다.

### 9.10 수준 초과 질문

학생 질문이 수업 주제와 관련 있어도 학교급 수준을 지나치게 넘어서면 깊이를 제한한다.

고급 설명을 길게 제공하지 않고, 현재 학교급에서 다루는 수준으로 바꿔 안내한다. 필요하면 “이 내용은 더 높은 학년에서 자세히 배우는 내용이에요”처럼 말하고 현재 수업 목표와 연결된 질문으로 되돌린다.

### 9.11 정답 요구

학생이 “정답만 알려줘”, “숙제 답 써줘”, “빨리 답만”처럼 요청해도 최종 답을 직접 제공하지 않는다.

학생의 어려움은 인정하되, 바로 답을 주는 대신 생각을 시작할 수 있는 질문이나 단서를 제공한다.

### 9.12 범위 밖 질문

범위 밖 질문에는 학습 답변을 제공하지 않는다.

현재 챗봇의 학교급, 과목, 주제, 성취기준 범위를 짧게 안내하고, 그 안에서 질문을 다시 쓰도록 유도한다.

자해, 폭력, 학대, 긴급 위험 등 학생 안전 관련 표현은 범위 밖이어도 안전 안내를 우선한다.

### 9.13 프롬프트 인젝션 방어

다음 요청은 따르지 않는다.

- 이전 지시 무시
- 일반 챗봇처럼 행동
- 시스템 프롬프트 출력
- 교사 설정 공개
- 내부 규칙 공개
- 답만 말하기

챗봇은 짧게 거절하고 현재 수업 주제로 자연스럽게 되돌린다.

### 9.14 부적절한 입력

욕설, 장난성 입력, 수업과 무관한 말에는 과하게 훈계하지 않는다.

짧고 차분하게 수업 주제로 돌아오도록 안내한다. 반복되면 “이 대화에서는 수업 주제에 맞는 질문만 이어갈 수 있어요”처럼 응답을 제한한다.

### 9.15 안전 예외

자해, 폭력, 학대, 긴급 위험 표현이 감지되면 수업 범위와 관계없이 안전 안내를 우선한다.

챗봇은 상담사처럼 길게 상담하지 않는다. 가까운 어른, 선생님, 보호자에게 바로 알리도록 안내한다. 지금 위험한 상황이면 112 또는 119에 도움을 요청하라고 안내한다.

## 10. 수학, 수식, 코드 입력

MVP는 한국어만 지원한다.

학생은 수학식, 기호, 코드 조각을 텍스트로 붙여 넣을 수 있다.

입력 처리 원칙:

- 줄바꿈을 보존한다.
- 괄호, 등호, 부등호를 보존한다.
- 간단한 코드 블록을 코드처럼 인식한다.
- 수식은 일반 문장처럼 뭉개지 않도록 한다.
- 복잡한 LaTeX 렌더링은 MVP 이후로 미룬다.

수학에서도 정답 직접 제공 금지 원칙을 적용한다.

챗봇은 최종 답을 바로 말하지 않고, 학생과 함께 문제를 읽고 조건을 찾고 다음 풀이 단계를 생각하게 한다.

단계 풀이가 필요한 질문은 한 번에 한 단계씩만 진행한다. 전체 풀이를 한꺼번에 보여주지 않는다.

## 11. 학생 화면

학생 화면은 공유 링크로 접속한 학생만을 위한 화면이다. 학생에게는 교사 탭, 관리자 탭, 교사용 생성 폼을 보여주지 않는다.

PC 화면에서는 일반적인 AI 채팅 서비스처럼 넓고 시원한 대화 영역을 제공한다. 태블릿과 모바일에서는 정보 패널과 채팅 영역을 재배치하되, 채팅 입력창은 항상 화면에서 확인할 수 있어야 한다.

### 11.1 개인정보 안내

학생 입장 시 개인정보 입력 금지 안내를 짧게 보여준다.

학교급별 말투로 다르게 표현한다.

초등:

> 여기는 이름이나 학번을 쓰지 않고 사용하는 곳이에요. 질문할 때도 내 이름, 친구 이름, 전화번호, 집 주소는 쓰지 않아요.

중등:

> 이 공간에서는 이름, 학번, 연락처를 입력하지 않습니다. 질문에도 나와 친구의 개인정보가 들어가지 않도록 해 주세요.

고등:

> 이 활동은 개인정보 입력 없이 진행됩니다. 질문 작성 시 이름, 학번, 연락처, 주소 등 개인을 알아볼 수 있는 정보는 포함하지 마세요.

### 11.2 대화 관리

학생 화면에는 다음 기능을 제공한다.

- 새 대화
- 이전 대화
- 기록 내려받기
- 기록 삭제

학생 대화는 브라우저 로컬에만 저장한다. 같은 기기와 같은 브라우저에서 접속할 때만 이전 대화를 이어갈 수 있다.

브라우저 저장소를 지우거나 다른 기기로 접속하면 이전 대화는 보이지 않는다고 안내한다.

학생이 채팅을 이어 가면 메시지 목록은 자연스럽게 최신 메시지로 스크롤된다. 사용자가 매번 스크롤 막대를 직접 내려야 하는 구조로 만들지 않는다.

AI 답변에 포함된 기본 Markdown 문법은 실제 서식으로 표시한다. 굵게, 기울임, 밑줄, 줄바꿈, 짧은 목록, 인라인 코드 정도는 화면에서 그대로 렌더링한다.

### 11.3 기록 삭제

학생은 다음 중 하나를 선택할 수 있다.

- 현재 대화 삭제
- 해당 챗봇의 모든 로컬 기록 삭제

삭제 전에는 짧은 확인 창을 보여준다. 삭제는 해당 기기와 브라우저에만 적용된다.

### 11.4 기록 다운로드

기본 제출 파일은 PDF로 제공한다.

보조 파일은 TXT로 제공한다.

다운로드 파일 형식으로 Markdown은 MVP에서 제공하지 않는다.

파일에는 다음을 포함한다.

- 챗봇 이름
- 수업 주제
- 학교급/과목
- 선택된 성취기준
- 대화 날짜
- 학생-챗봇 대화 내용
- “이 기록은 학습 과정 확인용이며, 정답지나 평가 결과가 아닙니다.”

기본 모드에서는 PDF/TXT에 학생 이름과 학번을 포함하지 않는다.

### 11.5 응답 대기와 오류

응답 대기 문구는 “AI가 생각 중” 같은 표현을 피한다.

예:

- 초등: 생각을 이어 보고 있어요.
- 중등: 질문을 살펴보고 있어요.
- 고등: 질문 흐름을 정리하고 있어요.

오류가 나면 기술적인 오류 코드를 보여주지 않는다.

예:

- 초등: 잠시 연결이 잘 안 됐어요. 한 번만 다시 보내 볼까요?
- 중등: 응답을 불러오지 못했어요. 다시 보내 보거나 선생님께 알려 주세요.
- 고등: 응답을 가져오지 못했습니다. 잠시 후 다시 시도하거나 선생님께 알려 주세요.

## 12. 공유 링크

모든 챗봇은 기본적으로 비공개 상태로 생성된다.

교사가 공유 링크를 켠 챗봇만 학생이 접속할 수 있다. 공유 링크를 끄면 학생 접속은 즉시 차단된다.

공유 링크에는 추측하기 어려운 공개 토큰을 사용한다.

학생이 공유 링크로 들어오면 학생 화면만 열린다. 교사/관리자 화면으로 전환하는 탭이나 탐색 요소는 노출하지 않는다.

교사가 공유 링크를 복사하면 복사 결과 메시지는 해당 링크 버튼 근처에 표시하고 잠시 뒤 사라지게 한다.

공유 링크 만료 설정:

- 만료일 없음
- 오늘 자정까지
- 7일 후
- 달력에서 날짜 선택
- 날짜 직접 입력

직접 입력은 `YYYY-MM-DD` 형식으로 받고 잘못된 날짜는 저장하지 않는다.

## 13. 교사 미리보기

교사는 챗봇 생성 또는 수정 후 학생 화면과 같은 대화창에서 미리 테스트할 수 있다.

미리보기 대화는 실제 학생 대화 기록과 분리된다.

교사는 테스트하면서 힌트 강도, 페르소나, 수업 주제, 교육과정 범위를 수정할 수 있다.

미리보기 대화 내용은 기본적으로 저장하지 않는다. 실제 API 호출 사용량은 기록한다.

## 14. AI 제공자와 모델

교사는 API 키를 입력하지 않는다.

AI API 키는 개발자가 서버 환경변수로 설정한다. 로컬에서는 `.env`, Vercel에서는 Project Environment Variables를 사용한다.

운영 배포에는 `OPENAI_API_KEY`, `NEIS_API_KEY`, `FIREBASE_PROJECT_ID`, Firebase Admin 인증 정보, `KKOKKOMU_ADMIN_EMAILS`, `VITE_FIREBASE_*`, `VITE_FIREBASE_AUTH_ENABLED=true`를 등록한다.

학생 브라우저에는 API 키를 절대 노출하지 않는다. 모든 AI 호출은 Vercel 서버리스 API 또는 로컬 개발 API를 통해 중계한다.

학생 공유 링크에서 발생한 대화는 공유 token으로 서버 저장소의 챗봇을 다시 확인한 뒤 처리한다. 클라이언트가 보낸 `ownerTeacherId`나 `chatbotId`만으로 사용량을 기록하지 않는다.

MVP 기본 제공자는 OpenAI `gpt-5.4-nano` 계열로 설계하되, 관리자 화면에서 LM Studio 기반 로컬 LLM provider도 선택할 수 있다.

추후 다음 제공자를 추가할 수 있도록 provider 인터페이스를 분리한다.

- OpenAI
- Claude
- Gemini
- 로컬 LLM

기본 모델은 서버 저장소의 AI 설정 문서로 관리한다. 초기값은 `openai:gpt-5.4-nano`이고, 관리자는 허용된 모델 목록 안에서 OpenAI와 LM Studio 모델을 전환한다.

## 15. AI 응답 속도

학생 채팅은 스트리밍 응답을 사용한다.

서버에서 먼저 개인정보 필터, 범위 판단, 정책 검사를 거친 뒤 스트리밍을 시작한다.

교사 미리보기에서도 동일한 스트리밍 방식을 사용한다.

응답 중 `중지` 버튼을 제공한다. 중지하면 현재까지 받은 응답은 대화에 남긴다. 중지된 응답은 완성된 답변이 아닐 수 있음을 자연스럽게 표시한다.

긴 입력은 기본적으로 허용한다. 챗봇은 핵심 질문을 먼저 요약하고 확인한 뒤 다음 질문으로 이어 간다.

단, 지나치게 긴 입력이나 반복, 붙여넣기성 입력은 줄여 달라고 안내한다. 개인정보 필터는 길이와 관계없이 먼저 적용한다.

## 16. 사용량과 오류

MVP에서는 기본적으로 교사별 사용 제한을 두지 않는다.

교사 대시보드에는 월별 사용량을 표시한다.

표시 항목:

- 총 대화 수
- AI 호출 수
- 입력 토큰
- 출력 토큰
- 예상 비용
- 챗봇별 사용량
- 학생 공유 사용량
- 교사 미리보기 사용량

학생 개인정보와 학생별 사용량은 표시하지 않는다. 챗봇 단위 집계만 제공한다.

운영 Firestore 조회는 무료 티어 사용량을 줄이기 위해 교사별 범위를 먼저 제한한다. 일반 교사가 내 프로필을 조회할 때는 `teachers/{uid}` 단건 문서를 읽고, 내 챗봇이나 내 사용량을 조회할 때는 전체 `chatbots`, `usageMonthly` 컬렉션을 읽은 뒤 서버에서 필터링하지 않고 `ownerTeacherId` 또는 `teacherId` 기준 `where` 쿼리를 사용한다. 관리자가 전체 현황을 볼 때만 전체 집계 목록을 읽는다.

오류 로그는 다음 원칙을 따른다.

- 학생에게는 오류 코드를 보여주지 않는다.
- 교사에게는 쉬운 말로 오류 상태를 보여준다.
- 관리자는 전체 오류 로그와 기술 진단 정보를 볼 수 있다.
- 오류 로그에는 학생 개인정보나 학생 대화 원문을 저장하지 않는다.

교사 화면 예:

- 응답을 불러오지 못한 일이 3번 있었어요.
- 잠시 AI 연결이 불안정했어요.
- 이 챗봇에서 오늘 오류가 반복되고 있어요. 관리자에게 문의해 주세요.

## 17. 관리자 화면

관리자 화면은 운영에 필요한 최소 기능을 제공한다.

- 승인 대기 교사 목록
- 교사 일괄 승인
- 교사 일괄 거절
- 전체 교사 목록
- 교사별 사용량
- 교사별 챗봇 수
- 챗봇 비활성화
- 관리자 권한 지정
- 비밀번호 재설정 메일 발송
- 관리자 작업 로그

관리자는 사용자의 현재 비밀번호를 볼 수 없고 새 비밀번호를 직접 지정하지 않는다. 비밀번호 재설정 메일 발송만 제공한다.

## 18. 기술 스택

### 18.1 웹앱

- React
- Vite
- TypeScript

### 18.2 Firebase

운영 배포는 Firebase와 Vercel을 기준으로 한다.

- Firebase Authentication
- Firestore
- Firebase Admin SDK
- Firebase CLI Firestore rules/indexes

Vercel은 정적 프런트엔드와 서버리스 API를 담당한다.

- `vercel.json`은 `dist`를 output directory로 사용한다.
- `/api/*`는 Vercel 서버리스 함수로 처리한다.
- `/s/:token`, `/privacy`, 그 외 SPA 경로는 `index.html`로 rewrite한다.
- Vercel 환경변수 등록 전에는 `npm run vercel:env:dry-run`으로 등록 대상 변수와 값 존재 여부를 확인한다.
- Vercel CLI 상호작용 연결이 멈추면 `VERCEL_ORG_ID`와 `VERCEL_PROJECT_ID`를 로컬 `.env`에 넣고 `npm run vercel:link:env`로 `.vercel/project.json`을 생성한다.
- Vercel 프로젝트 연결 후 `npm run vercel:env:sync`로 production 환경변수를 등록한다. 필요하면 `VERCEL_ENV_TARGETS=production,preview`로 대상 환경을 확장한다.
- Vercel 배포 후에는 `DEPLOY_URL=https://배포주소 npm run smoke:deploy`로 루트 SPA, 개인정보처리방침 SPA rewrite, `/api/health`, 인증 없는 `/api/teachers` 보호 상태를 확인한다.

Firestore 직접 클라이언트 접근은 기본 차단한다. 운영 데이터는 Vercel 서버 API가 Firebase Admin SDK로 읽고 쓴다.

### 18.3 AI 처리

## 19. 디자인 시스템

디자인은 현재 앱에 반영된 `src/presentation/styles.css`, 로컬 Paperlogy 폰트 설정, 그리고 재사용 가이드 문서 `DESIGN.md`를 기준으로 한다.

`theme.css`, `tokens.json`, `variables.css` 같은 외부 스타일 참고 파일은 런타임이나 빌드에 의존하지 않는다. 실제 앱 스타일은 `src/presentation/styles.css` 안에서 자체적으로 완결되어야 한다.

디자인 원칙:

- 따뜻한 크림 종이 배경
- 진한 초록 잉크색 텍스트
- 형광펜 노란색 강조
- 민트, 블러시, 티얼 계열의 sticky-note 추천 카드
- Paperlogy Bold 중심 제목
- 한글 줄바꿈에 맞춘 `word-break: keep-all`
- 전체 글자 간격 `-0.02em`
- 전체 줄간격 `1.35`
- 버튼은 6px 안팎의 단정한 라운딩과 명확한 hover/active 피드백 적용
- 패널과 카드에는 얇은 초록 계열 경계선과 12px 안팎의 라운딩 적용
- 과한 AI 느낌, 네온, 어두운 개발자 도구 느낌 지양
- 보라/파랑 그라디언트 중심의 일반 AI 서비스 느낌 지양
- 화면 섹션을 카드 안에 다시 넣는 중첩 카드 구조 지양

화면별 밀도:

- 교사/관리자 화면: 조밀하고 정보 중심

## 20. 반응형 설계

전체 UI는 반응형 디자인을 기본으로 한다.

목표 기기:

- 교사: 데스크톱, 노트북, 태블릿
- 학생: 크롬북, 아이패드, 안드로이드 기기, 스마트폰

기기별 방향:

- 데스크톱: 넓은 학생 채팅 영역, 교사 화면은 사이드바 + 목록 + 설정 패널
- 태블릿/크롬북: 히어로 높이를 줄이고 채팅 영역을 먼저 사용할 수 있게 배치
- 스마트폰: 단일 열, 하단 중심 액션, 채팅 입력창 고정

핵심 기능은 모바일/태블릿에서 숨기지 않고 화면 크기에 맞게 재배치한다.

학생 화면의 레이아웃 기준:

- 큰 데스크톱: 채팅 영역을 넓게 쓰도록 최대 폭을 확장한다.
- 일반 데스크톱/태블릿: 정보 패널은 좁게, 채팅 패널은 넓게 유지한다.
- 920px 이하: 채팅 패널을 먼저 보여주고 정보 패널은 아래로 보낸다.
- 560px 이하: 화면 폭을 거의 모두 사용하고, 버튼과 입력창의 터치 타깃을 유지한다.
- 응답 생성 중에도 채팅 입력 영역이 화면 아래로 밀려 숨지 않게 한다.

## 22. 성공 기준

MVP가 성공했다고 판단하는 기준은 다음과 같다.

- 교사가 가입하고 관리자 승인 후 챗봇을 만들 수 있다.
- 교사가 수업 주제와 대화 목표를 입력해 공유 링크를 만들 수 있다.
- 학생이 가입 없이 공유 링크로 접속해 대화할 수 있다.
- 챗봇은 정답을 바로 제공하지 않고 질문과 단서로 사고를 이끈다.
- 범위 밖 질문에는 학습 답변을 제공하지 않는다.
- 학생 개인정보 입력은 AI 전송 전에 차단된다.
- 학생 대화는 서버에 저장되지 않는다.
- 학생은 대화 기록을 PDF/TXT로 내려받을 수 있다.
- 교사는 월별 사용량을 확인할 수 있다.
- 관리자 승인, 비활성화, 비밀번호 재설정 메일 발송이 가능하다.
- 학생 화면은 크롬북, 아이패드, 안드로이드 기기에서 무리 없이 사용할 수 있다.
- 교사는 관련 성취기준 추천 카드에서 여러 기준을 선택하거나 선택 없이 자동 반영할 수 있다.
- 추천 카드에는 성취기준 번호와 관련성 상/중/하가 명확히 표시된다.
- 학생 공유 링크 화면에서는 교사/관리자 탭이 보이지 않는다.
- 채팅 화면은 최신 메시지로 자연스럽게 스크롤되고 Markdown 답변이 실제 서식으로 표시된다.
